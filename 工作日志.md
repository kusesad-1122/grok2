# 工作日志 —— Grok Build 安卓移植

本日志记录关键假设、A/B 路线决策,以及为把桌面 Rust 工程交叉编译到 Android 所做的全部补丁。

## 0. 用户指令与路线决策

- 任务书原文给出两条路线:A(交叉编译真二进制 + 终端宿主,最忠实)与 B(Kotlin 原生重写,兜底)。
- **用户中途明确指令:只走路线 A,禁止路线 B(“直接用我这个空仓库触发编译”“不准用路线 B”)。**
- 因此本项目全程按**路线 A**推进:把 `xai-org/grok-build` 的真实二进制交叉编译到
  `aarch64-linux-android`,用内嵌终端模拟器运行,`grok2` 空仓库承载 Android 宿主工程。

## 1. 环境与勘察(事实复核)

- 源码 `grok-build`:Apache-2.0;主二进制 crate `xai-grok-pager-bin`,产物名 `grok`;
  含 provider 抽象(anthropic/openai/openrouter/... 均出现),模型 `grok-4.5/grok-4/grok-3` 等。
- provider 配置真实存在:`~/.grok/config.toml` 的 `[model.<name>]` 支持
  `model / base_url / api_key / env_key / api_backend / extra_headers`,
  `api_backend ∈ {chat_completions, responses, messages}`——**messages 即 Anthropic 原生协议**。
  这让“多厂商 + 两种协议”无需自研,直接由真二进制承担。
- 环境:JDK 21、Rust 1.92.0;Android SDK/NDK 初始缺失,已安装
  (cmdline-tools、platform-34、build-tools 34.0.0、platform-tools、NDK 27.2.12479018)。
  `protoc` 经 apt 安装(3.21.12)。

## 2. 交叉编译工具链

- `.cargo/config.toml` 追加 `aarch64-linux-android` / `armv7-linux-androideabi` 目标的 NDK clang 链接器。
- 构建命令:`cargo build -p xai-grok-pager-bin --target aarch64-linux-android --no-default-features`。
  - `--no-default-features` 关闭 `jemalloc`(tikv-jemalloc 在 Android 上不适用)与
    `sandbox-enforce`(landlock/seatbelt 内核沙箱在移动端无意义;沙箱后端在 android 本就不编译)。

## 3. Android 交叉编译补丁(逐个)

Android 的 `target_os` 是 `"android"` 而非 `"linux"`。大量桌面向 crate 只识别 linux/macos,
交叉编译时在 android 分支“落空”。策略:能用 `[patch.crates-io]` 覆盖的用本地补丁副本
(集中于 `grok-build/third_party/android-patches/`),grok 自有代码直接改源码。**所有改动仅影响
android target。**

1. **nono 0.53.0**(landlock/seatbelt 沙箱库):`capability.rs` 的去重逻辑只为 linux/macos 定义
   `let key`,android 两分支皆不命中导致 E0425。补丁新增 `not(any(linux,macos))` 分支(按 linux 语义
   取 `resolved`)。nono 本就为“非 linux/非 macos”平台准备了 no-op 沙箱后端(`sandbox/mod.rs`),
   android 天然落入,`landlock` crate 也只在 `cfg(linux)` 作为依赖,故无需其它改动。

2. **waitpid-any 0.3.0**:仅 linux/bsd/windows 有 `imp` 模块。补丁为 android 新增 `android.rs`,
   用 libc 的 `SYS_pidfd_open` + `poll` 实现子进程等待(与 linux.rs 语义一致;rustix 未对 android
   导出 pidfd,故直接走 libc)。并为该补丁加 android-only 的 `libc` 依赖。

3. **mid 4.0.0**(机器 ID):把所有 `target_os = "linux"` 门控扩展为
   `any(linux, android)`——linux 实现通过 shell 读 `/etc/machine-id`,在 android 可编译,
   运行时若无该文件则优雅返回 Err。

4. **arboard 3.6.1**(剪贴板):上游平台选择**显式排除** android 且无 android 后端。
   补丁新增 `platform/android.rs`:进程内内存剪贴板(文本 set/get 可回环,供 grok 内部复制/粘贴;
   图片/文件/HTML 降级为不可用)。App 宿主另经 `TerminalSessionClient` 桥接系统剪贴板。

5. **grok 自有代码 · `xai-file-utils::workspace_classifier`**:`is_platform_home_excluded` 有
   macos/linux/windows 三个 cfg 变体但漏 android。将 linux 变体扩为 `any(linux, android)`
   (home 目录语义一致)。

6. **grok 自有代码 · `xai-grok-pager-render::theme::system_appearance`**:`dark_light::detect()`
   在 android(无桌面外观 API)返回 `Mode` 而非 `Result`,原 `match { Ok(..) }` 不匹配。
   新增 `#[cfg(target_os="android")]` 变体按值匹配,原变体加 `not(android)`;系统外观交 OSC 11 兜底。

> 里程碑:上述补丁后 `cargo check -p xai-grok-pager-bin --target aarch64-linux-android
> --no-default-features` 全绿(CARGO_EXIT=0),即整个 workspace 对 android 通过类型检查。
> 随后进行真正的 `cargo build`(codegen + 链接)。

## 4. Android 宿主工程(grok2)

- Kotlin + 经典 View + ViewBinding;`com.grokbuild.terminal`。
- 打包策略:交叉编译产物重命名为 `libgrok.so` 放入 `jniLibs/arm64-v8a/`,
  运行时从 `nativeLibraryDir` 直接 `exec`(规避 Android 10+ 的 W^X 执行限制)。
- 终端:vendored Termux `terminal-view` + `terminal-emulator`(含 `libtermux` PTY,ndkBuild 编译)。
- `GrokLauncher`:据激活 provider 生成 `$GROK_HOME/config.toml`,组装 env(HOME/GROK_HOME/PATH/
  TERM=xterm-256color/关闭遥测等),`TerminalSession(shellPath, cwd, args, env, ...)` 启动 grok;
  勾选 root 时以 `su -c "exec libgrok.so"` 启动。
- UI 全中文;groknight 真实配色(`colors.xml`);状态行显示 `厂商·模型·Root`;
  底部触屏辅助键(ESC/TAB/CTRL/ALT/方向键/^C^D^L 等)。
- 设置页:Spinner 选厂商,自动带出默认 base_url/model(可编辑),各厂商 Key 分别保存;
  Anthropic 自动用 `api_backend=messages` + `x-api-key`/`anthropic-version` 头。
- Root:libsu 检测/请求;无 root 不崩,降级为普通用户运行。

## 6. 假设与已知取舍

- **登录**:移动端无浏览器 OAuth,凭据完全走 API Key / config.toml;已设 `GROK_FOLDER_TRUST=0`
  等环境变量减少交互阻塞。
- **Anthropic 鉴权**:按官方文档用 `extra_headers` 写 `x-api-key` + `anthropic-version`,
  未设 `api_key`(避免与 Bearer 冲突);同时注入 `ANTHROPIC_AUTH_TOKEN` 供 env_key 回退。
- **沙箱/jemalloc**:移动端关闭,不影响 agent 与工具循环功能。
- 记录见下节 “验证” 与任何 `[UNVERIFIED]` 标注。

## 7. 验证(能真正失败的检查)

- **交叉编译**:`cargo build -p xai-grok-pager-bin --target aarch64-linux-android
  --no-default-features` → **CARGO_EXIT=0**;产物 `xai-grok-pager` 587MB(debug)。
- **ELF 校验**:`file` → `ELF 64-bit LSB pie executable, ARM aarch64, interpreter
  /system/bin/linker64`。`llvm-strip` 后 214MB,仍为合法 stripped PIE。
- **动态依赖**(`llvm-readelf -d`):NEEDED = liblog / libOpenSLES / libz / libdl / libm /
  libc / libc++_shared。前六个是 Android 系统库(设备必有),libc++_shared.so 随 APK 打包进
  `jniLibs/arm64-v8a`(运行时经 `LD_LIBRARY_PATH=nativeLibraryDir` 定位)。**依赖均可满足。**
- **APK 打包**:`./gradlew :app:assembleDebug` → **BUILD SUCCESSFUL**;`app-debug.apk` 73MB。
- **aapt dump badging**:package `com.grokbuild.terminal`;label `Grok Build`;
  launchable-activity `com.grokbuild.terminal.MainActivity`;权限 INTERNET + ACCESS_NETWORK_STATE。
- **native 库在 APK 内**:`lib/arm64-v8a/{libgrok.so, libtermux.so, libc++_shared.so}` 均在。
- **签名**:`apksigner verify` → **Verifies**(v2 方案,1 签名者,调试签名,可侧载)。

## 8. 自我批评(怀疑审阅者视角)与已知弱点

- **[UNVERIFIED · 最重要] 未在真机/模拟器执行**:本构建环境为 x86_64、无 `/dev/kvm`、无虚拟化,
  且 Android 程序解释器 `/system/bin/linker64` 仅存在于设备,NDK 只提供链接用 stub,故无法在此
  真正运行该 arm64 二进制(qemu 冒烟测试因缺 linker64 无法进行)。以上校验证明“能加载运行”的
  必要条件(合法 ELF + 依赖可满足 + 正确打包签名),但 grok TUI 在真机的实际交互/联网**未经真机验证**。
  建议:装机后打开即验证;若 grok 首启走浏览器登录界面,填好 API Key 的 config.toml 应使其走 API Key 路径。
- **[UNVERIFIED] Anthropic 鉴权细节**:按官方文档用 `extra_headers` 注入 `x-api-key` +
  `anthropic-version`,并额外设 `ANTHROPIC_AUTH_TOKEN` 环境变量供 env_key 回退;未设 `api_key`
  以避免与 Bearer 冲突。真机对 `api.anthropic.com` 的实际 401/200 未验证。
- **[UNVERIFIED] 首启登录流**:原 grok 默认浏览器 OAuth;移动端靠 config.toml 的 api_key 绕过。
  若某些代码路径仍强制交互登录,可能需要在 TUI 内跳过。
- **体积**:libgrok.so(strip 后 214MB,APK 内压缩至 ~70MB)首次安装解压较大;可后续用
  `--release` + `opt-level="z"` + LTO 显著瘦身(本次为节省编译时间用 debug)。
- **root 运行**:`su -c "exec libgrok.so"` 路径依赖设备 su 语义;不同 root 方案(Magisk 等)行为
  可能差异,无 root 时已降级为普通用户,不崩溃(已在代码中处理)。
